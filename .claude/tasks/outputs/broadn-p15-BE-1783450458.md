## Correction 1 — longitude sign-fix (`correct_coord()`)

- Positive longitude → negated, `coord_corrected=true`; if the negated point
  falls outside the continental-US bounding box (lat 24..50, lon -125..-66)
  it's still flagged `coord_corrected=true` but treated as
  `fetch_status="skipped_bad_coord"` (never queried).
- Null/NaN lat or lon → `coord_corrected=false`, `fetch_status="skipped_bad_coord"`.
- Result on the real dataset: 402 positive-lon rows, ALL negate cleanly into
  US bounds (0 corrected-but-still-bad). `coord_corrected_count=402`.

## Correction 2 — window aggregation (`classify_window()` + friends)

Four mutually-exclusive tiers (no_date evaluated first): `no_date` →
`window_exact` (date+time+duration>0, `duration_source="measured"`) →
`window_assumed_24h` (date+time, duration missing/0.0,
`duration_source="assumed_24h"`) → `date_only` (date, no time,
`duration_source="none"`, window = full calendar day).

- `compute_fetch_date_range()` resolves the minimal calendar-day span needed
  to cover `[window_start, window_end)` in ONE archive request (with a
  half-open-midnight edge case so a date_only window's exact-midnight end
  doesn't pull an unneeded extra day).
- `select_window_indices()` picks exactly the hourly points inside
  `[window_start, window_end)`; `aggregate_covariates()` is a single shared
  aggregation-rule function (precip SUM, wind_direction circular mean,
  others mean, temp min/max) reused for BOTH the window `covariates` and
  the calendar-day `covariates_daily` (DRY — no duplicated aggregation logic).
- Unit conversion (km/h→m/s, hPa→kPa) applied exactly once per fetched key,
  before any aggregation; cache persists RAW (unconverted) hourly arrays.
- Cache key is now `(round(lat,2), round(lon,2), start_date, end_date)` —
  a 2-day window is a distinct cache entry from a 1-day window at the same
  site. Added a prune step (filters the cache to the current run's
  `unique_keys` before `save_cache`) so orphaned p14-format single-day
  entries don't accumulate in the committed cache.

## IMPORTANT anomaly vs. task brief: "~90 recovered" is overstated

The task brief's Correction 1 says "this recovers ~90 previously-skipped
dated samples." Verified directly against the xlsx (independent of my
script, via plain pandas boolean masks):

- Old p14 `skipped_bad_coord` = 90 (dated rows with bad coord). This equals
  **77 null-coordinate rows** (unfixable — `Latitude`/`Longitude` both NaN)
  **+ 13 positive-longitude rows** that also had a date.
- Of the 402 total positive-longitude rows, **389 are `no_date`** — the sign
  fix doesn't help these at all since they were never going to be enriched
  regardless of coordinate validity.
- So the sign-fix actually recovers only **13** previously-skipped *dated*
  samples, not ~90. The 90 figure appears to conflate the total old
  skipped_bad_coord bucket (77 null + 13 pos-lon) with the recoverable
  subset. `data/covariates.json`'s new `skipped_bad_coord=77` is exactly
  the permanently-unfixable null-coordinate remainder — this is expected
  and correct, not a bug.
- Recommend PM/PI adjust any downstream copy referencing "~90 recovered" to
  "13 recovered / 77 permanently unenrichable (null coordinates)."
